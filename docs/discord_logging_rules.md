# VERDICT Discord Logging Rules

Operational rulebook for all Discord server logging performed by the VERDICT operator. All events generated by the Operations Project, and events flowing in from other VERDICT workflows, follow the logging rules codified in this document.

This document works in combination with:

- `qa_protocol_v1-1.md` — QA REVIEW block outputs are logged to the Operations Project session log, which may be summarized into Discord entries for high-severity outcomes
- `confirmed_strategic_decisions.md` — Discord channel structure (five channels) is fixed there; this document defines the operational rules that apply within that structure
- `vendor_notification_template.md` — vendor correspondence events log to `#vendor-responses` per the format defined here
- `media_pitch_template.md` — media engagement events log to `#media-outreach` per the format defined here
- `linkedin_posting_rules.md` — LinkedIn posting events log to `#daily-log` per the format defined here

If any rule here appears to conflict with `confirmed_strategic_decisions.md` (Discord channel structure), the confirmed decisions win. The five-channel structure is not modifiable at the Operations level; expansion requires internal approval.

---

## Why Discord logging matters

Discord logs serve three institutional functions that compound over time:

**Function 1: Audit trail.** Every external action VERDICT takes — vendor notifications, media pitches, LinkedIn posts, site updates — generates a log entry. The aggregate record across months and years becomes the operational history of VERDICT. This history is retrievable evidence in any future audit scenario: M&A due diligence, legal discovery, internal governance review, investor data room request.

**Function 2: Institutional memory.** Individual decisions and precedents erode from active memory within weeks. A logged record of why a decision was made, what response was received, and what next action was decided preserves the reasoning chain. Future Operations sessions (including sessions with co-founders or employees) can retrieve context that would otherwise be lost.

**Function 3: Pattern recognition.** Over time, logs reveal operational patterns: which journalists respond within 48 hours, which vendors send generic auto-replies, which incident types generate coverage, which LinkedIn post formats achieve engagement. These patterns inform strategic adjustments to templates and workflows. Without logs, pattern recognition relies on memory, which is unreliable at scale.

The five channels defined in `confirmed_strategic_decisions.md` partition these logs by operational concern. This partitioning is structural: it prevents `#daily-log` from becoming an undifferentiated stream that cannot support any of the three functions above.

---

## Core principles

1. **Log every external action.** If an action produces an effect outside the Operations Project session (a sent email, a published post, a repo commit, a site deployment), it generates a log entry. Internal drafts, brainstorming, and aborted actions do not require logs.
2. **Log in structured format.** Every log entry follows the channel-specific format defined in Section 4 (Log entry format standards). Freeform notes without structure are not logs; they belong in separate working files.
3. **Log in English.** All channel entries use English regardless of the language Operations is using in the working session. Discord logs may be read by future co-founders, auditors, or the operator himself reviewing months later; a single language simplifies retrieval and readability.
4. **Log verbatim when quoting.** When a log entry references external content (vendor response, journalist reply, news article), quote verbatim or omit entirely. Paraphrased quotations introduce interpretation, which defeats the audit-trail function.
5. **Log without credentials.** Passwords, API keys, authentication tokens, session cookies, signed URLs containing secrets, and any other credential material are never written to Discord, under any circumstance. This is non-negotiable because Discord server access is broader than Operations session access, and log entries are not encrypted at rest within Discord.
6. **Log promptly.** Log entries are written within the same Operations session that produced the external action, not deferred. Deferral introduces memory decay and violates Function 2 (Institutional memory) of the why section. Same-session logging is the default; end-of-session batch logging is an acceptable fallback but not preferred.

## Channel architecture

VERDICT operates five Discord channels for logging, as defined in `confirmed_strategic_decisions.md`. Each channel has a specific scope and event type range. Channel discipline — routing each event to exactly one correct channel — is what allows logs to support audit trail, institutional memory, and pattern recognition over time.

### Channel 1: `#daily-log`

**Scope:** Task completions, milestones, daily operational summaries, and execution events that do not belong to the more specialized channels below. `#daily-log` is the default channel for execution events that are Operations-generated and not specifically vendor-, media-, idea-, or link-related.

**Typical events logged here:**

- LinkedIn post published (URL in first comment confirmed, 60-second window met)
- Site update committed and deployed (getverdict.fyi HTML or platform page changes)
- Engine repo commit (evaluation generated, KNOWN_FACTS updated, framework version bumped)
- Evaluation prompt executed for a new platform (prompt filename, source engine version)
- QA Protocol FULL mode run completed for a non-vendor / non-media content piece (e.g., site copy, newsletter draft)
- Weekly milestone summary entries (end-of-week completion review)
- Any administrative activity with external effect (domain renewal, analytics configuration change, email alias added)

**Events that do NOT belong here:**

- Vendor-related events → route to `#vendor-responses`
- Media pitch / inquiry events → route to `#media-outreach`
- Strategic decisions or new rules → route to `#ideas`
- New URLs or reference materials → route to `#links`

**Edge case handling:**

- A LinkedIn post that mentions a specific vendor: log the post to `#daily-log` (execution event), and log any vendor response received to `#vendor-responses` (response event). These are two separate log entries.
- An evaluation publication (new platform evaluation added to getverdict.fyi): log to `#daily-log` as an execution event. If the vendor is subsequently notified via Template 1, log that notification to `#vendor-responses`.
- A site deployment that includes multiple changes (new evaluation + layout update + typo fix): a single `#daily-log` entry with a change summary is appropriate, not multiple entries.

### Channel 2: `#media-outreach`

**Scope:** All media engagement events — outbound pitches, inbound inquiries, journalist responses, and coverage outcomes. `#media-outreach` captures the full lifecycle of VERDICT's relationship with the journalism ecosystem, from first pitch to published coverage and beyond.

**Typical events logged here:**

- Media pitch sent (Templates 1–5 from `media_pitch_template.md`)
- Media inquiry received from a journalist (inbound initial contact)
- Inquiry response sent (Template 6, with Type I–IV classification)
- Journalist substantive reply received (not auto-reply or brief acknowledgement)
- Decline received from a journalist, with reasoning noted
- Firm decline received (journalist explicitly opts out of future outreach)
- Coverage published that cites VERDICT (with framework URL citation flag)
- Coverage published that uses VERDICT data without citation (tracked for pattern recognition, no intervention)
- Escalation triggered on a media event (routed for internal escalation; escalation timestamp logged)

**Events that do NOT belong here:**

- Vendor correspondence about evaluation disputes → `#vendor-responses` (even if the dispute originated from a journalist's published coverage)
- LinkedIn post publications → `#daily-log` (execution event)
- Strategic decisions about media strategy → `#ideas`
- New journalist contact information discovered → `#links`

**Edge case handling:**

- A journalist's reply that contains both a factual correction claim and a general inquiry: log the overall event to `#media-outreach` as "substantive reply received." Route the factual correction handling per `vendor_notification_template.md` Template 3 process (even though requester is a journalist), and log that correction processing to `#vendor-responses` separately.
- A journalist publishes coverage that misrepresents VERDICT data: log the publication to `#media-outreach` as "coverage published, misrepresentation flagged," and route the escalation for internal review. Do not draft a correction response in the log; the correction response itself (if approved) is logged as a subsequent entry.
- A journalist uses VERDICT data without citation but the coverage is accurate: log as "coverage published without citation, no intervention." Repeat omissions from the same journalist inform future pitch framing but do not trigger action.

### Channel 3: `#vendor-responses`

**Scope:** All vendor correspondence events — outbound notifications, inbound responses, correction requests, factual disputes, and escalation. `#vendor-responses` is the operational complement to `vendor_notification_template.md` and contains the full institutional record of VERDICT's communication with evaluated parties.

**Typical events logged here:**

- Vendor notification sent (Templates 1–4 from `vendor_notification_template.md`)
- Vendor response received (auto-reply, brief acknowledgement, substantive engagement, or factual correction request)
- Correction applied to published evaluation (with revision history entry reference)
- Correction declined (with reasoning category applied)
- Request for additional documentation sent (Template 3 variant)
- Security team reply sent (Template 4 from vendor notification template)
- Escalation triggered on vendor correspondence (routed for internal escalation)
- Vendor initiates contact proactively (vendor reaches out before any VERDICT notification)

**Events that do NOT belong here:**

- Media / journalist events → `#media-outreach` (even when journalist asks about a vendor-related topic; separate routing applies per edge cases)
- LinkedIn posts mentioning vendors → `#daily-log` (execution event, not vendor correspondence)
- Strategic decisions about vendor engagement policy → `#ideas`

**Edge case handling:**

- A vendor's reply is forwarded to press@ by mistake (vendor confused about the correct alias): log to `#vendor-responses` regardless of the receiving alias, with a note on the misrouting. The event is a vendor correspondence, and the routing error is incidental to the event type.
- A vendor threatens legal action: log the receipt of the communication verbatim to `#vendor-responses`, note the escalation-trigger flag, and handle per the internal escalation procedure. Do not include any Operations response in the log (there should not be one without Strategy approval).
- A vendor sends proactive outreach asking to be evaluated: log to `#vendor-responses` as "vendor-initiated contact," note that the VERDICT evaluation process does not accept requests for prioritization, and route any substantive discussion for internal review. No individual evaluation is scheduled based on vendor request alone.

### Channel 4: `#ideas`

**Scope:** Strategic decisions, operational pivots, new rules, insights derived from pattern recognition, and any institutional learning that affects how VERDICT operates going forward. `#ideas` is the channel where Operations-level observations get elevated to Strategy-level discussions, and where confirmed strategic decisions get their initial documentation before flowing into `confirmed_strategic_decisions.md`.

**Typical events logged here:**

- Strategic decision confirmed (new rule, policy change, or reversal of prior decision)
- Pivot event (change in direction for Phase 1 distribution, revenue phase, framework, or any core element)
- New operational rule created (e.g., "60-second window rule" for LinkedIn first comment, originating from post engagement observation)
- Insight derived from pattern recognition (e.g., "Vendors with VDP programs route security@ to security analysts — bypass with privacy@")
- Co-founder or external advisor input that materially informs an operational or strategic decision
- Failed experiment or lesson learned (e.g., "Template 4 incident pitch without multi-recipient disclosure generated exclusivity confusion — disclosure now mandatory")
- Governance observation (e.g., observation about boundary between internal review and Operations Project, feedback on template design)
- Anomaly noticed that may warrant future investigation (e.g., "Two journalists declined same week with similar reasoning — possible beat shift in enterprise security coverage")

**Events that do NOT belong here:**

- Execution events (task completion, publication, send confirmation) → `#daily-log`
- Vendor or media correspondence → `#vendor-responses` or `#media-outreach`
- New URLs or reference materials → `#links`
- Routine reminders or scheduled tasks → not logged; these belong in separate task management tools

**Edge case handling:**

- A decision is tentative and still under reflection: log to `#ideas` as "tentative observation, pending decision." The tentative flag distinguishes this from confirmed decisions and signals that the entry is a thinking record, not a binding rule. If the observation later becomes a confirmed decision, a second entry records the confirmation.
- A pattern is noticed across vendor responses (e.g., "Most OSS platforms respond to Template 1 within 72 hours; most enterprise platforms auto-reply and do not follow up"): log to `#ideas` as "pattern observation" rather than duplicating across `#vendor-responses`. Individual vendor events stay in `#vendor-responses`; the pattern meta-observation lives here.
- An insight contradicts an earlier `#ideas` entry: log the new insight with a reference to the earlier entry. Do not edit or delete the earlier entry; institutional learning includes the track of revised thinking.

### Channel 5: `#links`

**Scope:** New URLs, contact information, reference articles, and external resources discovered during Operations activity that may have future utility. `#links` is VERDICT's external-reference archive — not a bookmark list, but a curated record of sources that have been found useful or that warrant future consideration.

**Typical events logged here:**

- New journalist contact information discovered (name, publication, email, beat, verified source)
- New vendor contact alias discovered (e.g., privacy@ replacement for security@ routing)
- New VDP or disclosure program identified for a specific vendor
- Reference article discovered that is relevant to a future pitch, evaluation, or strategic discussion
- New data source identified (e.g., a newly-tracked CVE database, a vendor-specific security advisory feed)
- External research report or industry analysis that informs VERDICT's framework development or methodology
- Academic paper or public thesis on AI agent trust, evaluation frameworks, or related topics
- Tool or service discovered that may support VERDICT operations (e.g., a new analytics tool, a new email infrastructure provider under consideration)

**Events that do NOT belong here:**

- URLs of published VERDICT evaluations → these are not "external references"; they are internal outputs, logged in `#daily-log` under publication events
- URLs of confirmed vendor or journalist communications → these are correspondence, logged in `#vendor-responses` or `#media-outreach`
- Social media posts, podcast episodes, or ephemeral content → unless they specifically inform a future Operations action, these do not warrant logging

**Edge case handling:**

- A URL is relevant to both a future media pitch and a future vendor notification: log once to `#links` with both potential uses noted in the entry. Duplicate logging across channels creates retrieval noise.
- A URL becomes dead (HTTP 404 or equivalent) after being logged: do not delete the original entry. Add a follow-up entry noting "URL no longer resolves as of [date]" if the dead link has operational significance (e.g., a journalist's contact page that no longer exists). Otherwise, the original entry remains as historical record.
- An entry references a source that is later discovered to be inaccurate (e.g., a secondary news source misreported a vendor acquisition): add a correction entry referencing the original, do not edit or delete. This preserves the reasoning chain and the fact that VERDICT briefly considered the inaccurate source.

## Log entry format standards

Every Discord log entry follows a structured format. The common structure applies across all five channels; channel-specific field additions are defined in Section 5 (Content rules) and Section 6 (Channel routing matrix). Format adherence is what distinguishes a log entry from an unstructured Discord message.

### Common structure

Every log entry contains these four mandatory fields, in this order:

1. **Timestamp** (when the event occurred or when the log is written, whichever is earlier)
2. **Event type** (classification of what kind of event this is, drawn from the channel's event type enumeration)
3. **Key identifier** (the primary subject of the event: platform name, journalist name, vendor name, URL, or commit hash)
4. **Next action** (what is planned next, or "no action" if the event concludes without follow-up)

These four fields answer the essential questions for audit trail and retrieval: when, what type, about whom or what, and what follows.

### Timestamp format

Use ISO 8601 date format with JST time zone annotation:

```
2026-04-23 14:37 JST
```

Rules for timestamp:

- Always JST. Do not use UTC, local time of non-Japan residence, or 12-hour format. JST is VERDICT's operational time zone, and consistent timestamps simplify retrieval.
- Date and time separated by single space. Do not use "T" separator (e.g., `2026-04-23T14:37` is RFC 3339 but harder to read in Discord UI).
- Time in 24-hour format, colon-separated (`14:37`, not `2:37 PM` or `14.37`).
- Suffix `JST` makes the time zone explicit for future readers who may retrieve logs from different time zones (co-founders in other countries, auditors, etc.).
- For events that occurred at a specific time outside the current Operations session (e.g., an email received at 03:15 JST and logged at 10:00 JST the same day), use the **event time** (03:15) not the log time (10:00). Note the distinction: log entries record the event, not the logging act.

### Event type field

Each channel has a specific enumeration of permitted event types. Event type is written as a single enum value, in lower-case kebab-case, wrapped in brackets:

```
[pitch-sent-template-1]
[vendor-notification-sent-template-2]
[response-received-substantive]
[correction-applied]
[strategic-decision-confirmed]
[new-journalist-contact]
```

Rules for event type:

- Use the exact enum values defined in the channel's event type list (Section 6 Channel routing matrix).
- Do not invent new event types without routing the new type through internal review for approval.
- Kebab-case (hyphens between words) with all lowercase. Underscores and camelCase are not used.
- Wrap in square brackets so the event type is visually distinct from free-text content.
- If an event is genuinely novel and does not fit any existing enum value, use `[other-flag-for-review]` and include a description in the log body. internal review periodically reviews `[other]` entries to identify new event types worth adding to the enumeration.

### Key identifier field

The key identifier is the primary subject of the event. What counts as the key identifier depends on the event type:

- For vendor events: vendor name (e.g., "aider" or "Aider AI LLC")
- For media events: journalist name + publication (e.g., "Thomas Claburn / The Register")
- For publication events: URL or platform name (e.g., "getverdict.fyi/aider/" or "aider")
- For commits: short commit hash (e.g., "babe9ba")
- For reference events (`#links`): the URL or resource title

Rules for key identifier:

- Quote the identifier so it stands out (e.g., `"aider"`, `"Thomas Claburn / The Register"`)
- Use the canonical form: evaluated platforms use their lowercase name ("aider"), journalists use the byline form ("Thomas Claburn"), vendors use legal entity name when documenting corporate identity or platform name when documenting evaluation events.
- If the identifier is ambiguous (multiple "aider" projects exist), disambiguate with additional context (e.g., `"aider-chat (Aider AI LLC's terminal coding agent)"`).
- If there is no clear single identifier (e.g., a cross-platform insight in `#ideas`), use a short description instead (e.g., `"VDP-routed vendor pattern"`).

### Next action field

The next action field records what follows from this event, or explicitly states that nothing follows:

- `→ Follow-up in 14-21 days if no response`
- `→ Monitor #media-outreach for journalist response within 48h`
- `→ internal review escalation (opening message posted)`
- `→ Re-evaluate platform in 90 days`
- `→ No action required (event concludes)`

Rules for next action:

- Start with an arrow (`→`) for visual distinction from the event description.
- Keep it concrete and time-bounded when possible ("within 48h," "in 90 days," "at next weekly review").
- If no follow-up is planned, write `→ No action required (event concludes)` explicitly. Empty or missing next-action fields signal incomplete logging.
- If the next action depends on a future condition, state the condition ("→ If vendor responds with documentation, route to Template 3; otherwise, no action").

### Channel-specific format additions

Beyond the four mandatory common fields, each channel has specific additional fields that capture channel-relevant context. These additions are structured, not freeform, and are appended to the common format.

**`#daily-log` additions:**

- **Source system** (for commits, which repo: engine / site / other; for publications, which platform: LinkedIn / site / other)
- **Artifact URL** (if the event produced a retrievable artifact: post URL, commit URL, evaluation page URL)
- **Outcome marker** (`success` / `partial-success` / `needs-followup`) — explicit state of the execution event

**`#media-outreach` additions:**

- **Template used** (1 through 6, matching `media_pitch_template.md` Templates)
- **Journalist + publication** (full identifier form, e.g., `"Thomas Claburn / The Register"`)
- **Response window** (expected wait period before considering the event concluded, e.g., `14-21 days` for Template 1)
- **Citation flag** (only for coverage publication events: `framework-url-cited` / `evaluation-url-only-cited` / `no-citation`)

**`#vendor-responses` additions:**

- **Template used** (1 through 4, matching `vendor_notification_template.md` Templates)
- **Vendor name** (evaluation context form, e.g., `"aider"` — distinct from legal entity form used in ownership events)
- **Response classification** (for inbound events: `auto-reply` / `brief-acknowledgement` / `substantive-engagement` / `factual-correction-request` / `legal-threat`)
- **Correction outcome** (for correction events: `applied` / `declined` / `additional-documentation-requested`)

**`#ideas` additions:**

- **Decision status** (`confirmed` / `tentative` / `proposed` / `deprecated`)
- **Scope** (which operations this affects: `vendor-engagement` / `media-engagement` / `framework` / `platform-infrastructure` / `cross-operations`)
- **Linked sources** (optional: references to other `#ideas` entries, external sources, or observations that informed this entry)

**`#links` additions:**

- **Reference type** (matching the taxonomy in Channel 5 Scope: `journalist-contact` / `vendor-contact` / `vdp-program` / `reference-article` / `data-source` / `research-report` / `academic-paper` / `tool-service`)
- **Intended use** (why this is logged: which future Operations action this may inform)
- **Verification date** (when the URL or contact was last confirmed accessible / accurate)

## Content rules

Content rules govern what is written in log entries and what is deliberately excluded. Format standards (Sections 4) define the structure; content rules define the substance. Both apply to every log entry.

Content rules are not aesthetic preferences. They are derived from the three functions Discord logs serve (audit trail, institutional memory, pattern recognition) and from the privacy and legal considerations detailed in Section 8 (Retention and privacy).

### Inclusion rules: what log entries must contain

Every log entry must contain:

**1. All four common mandatory fields (Part 6):** timestamp, event type, key identifier, next action. Missing any of these makes the entry unusable for retrieval.

**2. All applicable channel-specific additions (Part 7):** for example, Template used for `#vendor-responses`, Response classification for inbound vendor events, Citation flag for coverage publication events. Omitting these reduces the entry's analytic value for pattern recognition.

**3. Sufficient context to be interpretable in isolation:** future retrievers (operator six months later, co-founder, auditor) may read a single log entry without reading surrounding entries. The entry must self-contain enough context. This does not mean verbosity — it means including the entity names, dates, and classifications that make the entry stand on its own.

**4. Verbatim quotations when referencing external content:** if a vendor response, journalist reply, or news excerpt is mentioned, the text is quoted exactly as received, not paraphrased. Principle 4 (Log verbatim when quoting) applies throughout. Paraphrased summaries are acceptable only for generic auto-replies and brief acknowledgements where verbatim quotation provides no additional information.

**5. References to relevant external artifacts:** if the event produced or related to an external artifact (published URL, commit hash, email thread, tracker entry), the reference is included. This supports retrieval workflows that begin in Discord and navigate outward to the referenced artifact.

**6. Explicit outcome or resolution marker:** each entry concludes with clear outcome language. "Event concluded with no action required," "Escalated to Strategy," "Routed to Template 3 for correction review," etc. Entries that end without resolution markers are incomplete and signal that the logging was deferred.

### Exclusion rules: what log entries must NOT contain

Never include the following in any Discord log entry:

**1. Credentials of any kind:**

- Passwords (plaintext, hashed, or redacted forms like "the password is ****")
- API keys (full, partial, or described in ways that could leak the key)
- Authentication tokens (OAuth, JWT, session tokens)
- Session cookies
- Signed URLs containing secrets (e.g., AWS pre-signed URLs with access credentials embedded)
- Recovery codes or MFA seeds
- Private SSH keys or certificate private material

Even referring to the existence of credentials in certain ways is inadvisable. Write "Authenticated via 1Password entry for [service name]" rather than "Used password X to log in." The distinction is that a breach of the Discord server should not yield any credential material.

**2. Personal data beyond what is necessary:**

- Do not log the home addresses, phone numbers, or personal email addresses of journalists, vendors, or external contacts unless the contact is professional (publication byline, company contact page). If a contact has shared personal information expecting confidentiality, do not log it even with the contact's implicit or explicit permission — Discord is not a privacy-controlled environment.
- Do not log personal observations about journalists or vendors that would be embarrassing or defamatory if seen by the subject (e.g., "Journalist X seems inexperienced"). All logged observations should be factual and professional.

**3. Paraphrased quotations:**

- Re-stated version of what a vendor, journalist, or external source said, rather than the verbatim wording. Paraphrasing introduces interpretation, which defeats audit trail integrity.
- Exception: generic auto-replies and brief acknowledgements may be summarized ("auto-reply acknowledging receipt," "one-line thanks with no substantive content") because verbatim quotation provides no additional information in these cases.
- Even with the exception, the summary must be factual and minimal. "Auto-reply acknowledging receipt" is acceptable; "Auto-reply that seemed dismissive" introduces interpretation and is not acceptable.

**4. Speculation or assumption presented as fact:**

- Do not log "Vendor will probably respond next week" as if it is a fact. If speculation is included (for planning purposes), mark it explicitly: "Prediction (not confirmed): Vendor may respond by next week based on past response patterns."
- Do not log outcomes that have not yet occurred as if they have. "Pitch sent — journalist likely to publish coverage" is speculation. "Pitch sent — monitoring for response within 14-21 days" is factual.

**5. Attack language, ad hominem, or unprofessional tone:**

- Do not log content with hostile language about vendors, journalists, competitors, or any external party. Examples to avoid: "Vendor ignored the obvious issue," "Journalist was too lazy to verify," "Competitor's methodology is garbage."
- Professional register in logs is not aesthetic preference. Discord logs can be discovered in legal proceedings; unprofessional content in logs damages institutional posture and may support adverse party's narrative framing.
- Factual critique is acceptable and different from attack language: "Vendor's response did not address the specific documentation request" is factual; "Vendor deliberately avoided the question" is attributing intent and is not acceptable.

**6. Content that violates blocklists defined in other Operations documents:**

- The Inflammatory language blocklist from `qa_protocol_v1-1.md` Pass 2 applies to Discord logs: never use terms like "negligent," "fraudulent," "deceptive," "reckless," etc. in reference to vendors, journalists, or external parties within log entries.
- The AI writing style blocklist from `qa_protocol_v1-1.md` Pass 3 applies: avoid "At the end of the day," "game-changer," "landscape is shifting," etc. These are style markers that reduce the institutional quality of the log.
- Log entries are internal-facing but not immune to these rules. External discoverability in legal or audit contexts means the same standards apply.

### Edge cases and tension resolution

Inclusion and exclusion rules sometimes create tension. The following scenarios describe how to resolve the tension in ways that preserve both audit trail integrity and privacy/legal protection.

**Scenario A: Vendor response contains sensitive claims**

A vendor responds to a VERDICT notification with a statement that, if quoted verbatim, contains allegations about third parties (e.g., "Our competitor is violating our IP, which is why we focus our security resources elsewhere"). Verbatim quoting is required (Principle 4 and Inclusion rule 4), but the third-party allegation creates legal exposure if the log is discoverable.

Resolution: Log the event with a verbatim quote of the portion relevant to VERDICT (the vendor's engagement with the notification itself), and use placeholder summarization only for the third-party allegation portion: `[third-party allegation redacted; full text in vendor@ mailbox thread, reviewed 2026-04-23]`. The principle is minimum redaction — preserve verbatim wherever possible, redact only the specific segment that creates exposure, and always point to the full text in a restricted-access location.

**Scenario B: Journalist reply reveals confidential information**

A journalist replies with context that reveals confidential information about their publication's editorial process or internal sources. Logging this verbatim could compromise the journalist if the log is ever discovered.

Resolution: Log the event as "substantive reply received" with the journalist's direct reply to VERDICT's pitch quoted verbatim, but omit or summarize any portion about the publication's internal editorial process or confidential sources. The journalist's expectation of confidentiality in professional correspondence takes precedence. If the relevant content cannot be separated from the confidential context, log only that "reply received, confidential content shared, details in press@ mailbox" and do not quote the reply.

**Scenario C: Internal operational error with external impact**

VERDICT sends a pitch with a factual error (e.g., wrong CVE number) to multiple journalists. The error is discovered after send. The full situation must be logged (inclusion requirement), but the logging itself could create adverse evidence if discoverable.

Resolution: Log the event with factual accuracy: `[pitch-sent-template-4-error-discovered] | "LiteLLM supply chain incident pitch batch" | Error identified: Wrong CVE number cited (CVE-2026-XXXXX was correct, CVE-2026-YYYYY was used). Correction advisory drafted per Template 4 post-dispatch error process. → Correction advisory sent to all recipients within 4 hours of error discovery.` The record is honest and complete. Attempts to hide the error through incomplete logging create worse exposure (revision history discrepancy) than the error itself.

**Scenario D: Observation could be interpreted as attack language**

Operations notices a pattern that is factually accurate but could read as attack language (e.g., "Four of the last five vendors in Category X have no published security policy"). The factual observation is valuable for pattern recognition (`#ideas`), but the phrasing risks being read as adverse framing.

Resolution: Frame the observation as a factual pattern without causal attribution. Acceptable: `[pattern-observation] | "Category X security policy publication rate" | Four of last five Category X vendors have no published security policy. Observation neutral; may inform future pitch framing or report selection. → Monitor through 2026-Q2.` The observation is logged factually without implying intent or judgment.

**Scenario E: Credential exposure needs to be logged but cannot be written**

A credential was accidentally shared (e.g., an API key was sent to a journalist in a pitch by mistake, or a vendor sent their credentials in a response). This is a critical incident that requires logging, but the credentials themselves cannot be in the log.

Resolution: Log the incident with complete context but without the credential material. Example: `[credential-exposure-incident] | "API key accidentally included in Template 1 pitch to [Journalist / Publication]" | Key rotated immediately upon discovery (2026-04-23 14:45 JST). Affected journalist informed of accidental send and asked to delete. New key deployed in Template 1 validation checklist to prevent recurrence. → Incident report filed with internal review; pattern review scheduled for Q2.` The credential itself never appears; the incident and remediation are fully documented.

### Balance principle: substance over caution

Inclusion and exclusion rules in tension should be resolved in favor of substantive logging wherever privacy/legal considerations permit. Over-cautious logging creates its own institutional harm:

- **Under-logged events** mean audit trail has gaps, which is worse than logged-but-redacted content.
- **Vague or summarized entries** when verbatim would be appropriate reduce institutional memory quality.
- **Missing context for interpretability** forces future retrievers to reconstruct events from fragmentary signals, which is unreliable.

The default is to log substantively. Exclusion rules create specific narrow carve-outs. If an exclusion rule does not clearly apply, include the content.

When in genuine doubt, Operations routes the logging question for internal review for judgment. Do not adopt over-caution unilaterally; the balance between substance and caution is a strategic-level calibration.

### Quality review

At the end of each operational week, Operations performs a quality review of that week's log entries:

- **Format compliance check:** every entry has all four mandatory fields and applicable channel-specific additions
- **Content compliance check:** no prohibited content (credentials, paraphrased quotations outside exceptions, attack language, etc.)
- **Sufficient-context check:** each entry is interpretable in isolation

The review is fast (10-15 minutes for a typical week of entries) and catches pattern drift before it accumulates. Corrections to past entries are made as follow-up entries, not edits to the original — Preserving the original entry is Principle 5 adjacent: the record of how Operations logged at the time is itself audit trail.

If the review identifies a systematic logging pattern that violates rules, a `#ideas` entry flags it for internal review review: "Observation: [specific logging pattern] has appeared in multiple weeks. Consider clarifying [rule] or updating Discord logging rules document."

### Cross-channel scenarios

Some events span multiple channels naturally. When an event has components that belong in different channels, Operations creates **separate log entries in each affected channel** rather than a single multi-channel entry. The separate entries cross-reference each other through timestamp proximity and shared identifier.

**Scenario 1: LinkedIn post mentions a vendor + vendor responds**

Three separate events, three separate log entries:

1. **LinkedIn post published**: `#daily-log` entry with `[post-published-linkedin]` event type. Subject mentions vendor by name.
2. **Vendor response received** (if vendor sees post and responds): `#vendor-responses` entry with `[vendor-response-*]` event type, depending on response classification.
3. **Subsequent action** (if any — correction, Template 4 security-team reply, escalation): separate log entry in appropriate channel.

The three entries share the vendor name as identifier, allowing retrieval by vendor across channels.

**Scenario 2: Incident pitch multi-recipient dispatch + responses**

One outbound event, multiple inbound events:

1. **Template 4 incident pitch dispatch**: `#media-outreach` entry with `[pitch-sent-template-4]`. Records full journalist list (e.g., 6 recipients).
2. **Response from Journalist A (substantive)**: `#media-outreach` entry with `[journalist-response-substantive]`. Routes to Template 6 process.
3. **Response from Journalist B (generic auto-reply)**: `#media-outreach` entry with `[journalist-response-generic]`. No-action flag.
4. **Coverage published by Journalist C**: `#media-outreach` entry with `[coverage-published-framework-cited]` (or variant based on citation quality).
5. **No response from Journalists D, E, F** within 72 hours: single `#media-outreach` entry with `[incident-pitch-cycle-closed]` noting the 3 non-responsive recipients and cycle closure per Template 4 Part 13 post-dispatch monitoring rules.

Six events generated from one dispatch. All share the incident name as identifier.

**Scenario 3: Factual correction from journalist (not vendor)**

A journalist writes to VERDICT with a factual correction claim supported by documentation. Per `media_pitch_template.md` Part 17 rules and `vendor_notification_template.md` Part 15 cross-template routing, the correction process uses vendor notification template structures even though the requester is a journalist.

Two separate log entries:

1. **Journalist inquiry received (correction request)**: `#media-outreach` entry with `[journalist-response-substantive]` event type, noting the correction component. This preserves the media channel record of the journalist interaction.
2. **Factual correction handling**: `#vendor-responses` entry with `[vendor-response-correction-request]` event type, tracking the correction process per Template 3 workflow. The requester is a journalist, not a vendor, but the correction workflow belongs to the correction-channel record.

Both entries share the evaluation URL or platform name as identifier. Future retrieval of "all corrections to aider evaluation" returns this correction regardless of whether the search begins from media or vendor channel.

**Scenario 4: New journalist contact + immediately-planned pitch**

Operations discovers a new journalist's contact information and immediately plans a cold pitch based on fit.

Separate log entries (not combined):

1. **New journalist contact identified**: `#links` entry with `[new-journalist-contact]`. Archive-quality record of the contact with verification date.
2. **Media pitch sent (Template 1 Cold)**: `#media-outreach` entry with `[pitch-sent-template-1]`, referencing the contact entry (but not duplicating its content).

The separate entries support different retrieval functions: `#links` for contact archive browsing ("who are the journalists VERDICT has identified?"), `#media-outreach` for pitch history ("what pitches has VERDICT sent?"). Combining them into one entry would damage both retrieval paths.

### Matrix maintenance

The channel routing matrix is not frozen. It evolves as Operations encounters new event types and as templates are updated. Maintenance protocol:

**Adding a new row:**

- Operations identifies a novel event source through `[other-flag-for-review]` entries
- After 2+ similar `[other]` events within a calendar quarter, Operations drafts a proposed new row in an `#ideas` entry with `[operational-rule-created]` event type
- internal review reviews the proposal, approves or adjusts
- Approved new row is added to this document's matrix tables at the next document revision
- Retroactive re-classification of existing `[other]` entries to the new event type is optional; preserving original `[other]` entries as historical record is also acceptable

**Modifying an existing row:**

- If a template changes (e.g., new variant added to Template 1), the corresponding matrix row may need adjustment
- Operations flags the discrepancy in an `#ideas` entry with `[governance-observation]` event type
- internal review reviews; document is updated in next revision

**Removing a row:**

- A row is removed only when the corresponding event type is formally deprecated (template retired, workflow discontinued)
- Removal is a strategic-level decision; Operations does not remove rows unilaterally
- Removed rows' historical log entries remain in Discord; they are not retroactively re-classified

**Matrix drift indicators:**

- Persistent `[other-flag-for-review]` entries that do not mature into new rows (indicates Strategy review backlog or ambiguous event types)
- Multiple rows with the same event source but different enum values (indicates template fragmentation; requires rule review)
- Events routinely logged to the wrong channel despite matrix availability (indicates matrix usability issue; requires Operations review)

If any drift indicator is observed for more than 1 quarter, the drift itself is logged to `#ideas` with `[governance-observation]` for Strategy review.

## Workflow integration

This section defines how Discord logging integrates with each of VERDICT's five operational documents. Each template or ruleset generates a specific set of events that require logging; the integration rules below ensure that Discord logging does not become a separate parallel activity but remains embedded in the normal workflow of each document.

The logging integration respects each document's own workflow steps and adds logging as the final step of that workflow. Operations does not finish a template's execution until the log entry is complete.

### Integration with vendor_notification_template.md

Vendor notification workflow has defined post-send actions (Part 14 of that document). Discord logging is step 3 of those actions. The integration specifics:

**When Template 1 (Standard) notification is sent:**

- After confirming send success (Post-send action 1) and archiving to vendor mailbox (Post-send action 2), Operations logs to `#vendor-responses` with `[vendor-notification-sent-template-1]` event type.
- Log entry fields: timestamp, event type, vendor identifier (platform name), notification details (recipient alias used, subject line), next action (expected response window).
- The log is written within the same Operations session as the notification send. Deferral violates Principle 6 (Log promptly).

**When Template 2 (VDP-routed) notification is sent:**

- Same workflow as Template 1, but event type is `[vendor-notification-sent-template-2]`.
- Log entry includes additional verification: which non-security alias was used (privacy@, dpo@, legal@, etc.) and why (VDP routing identification rationale, if not obvious).

**When Template 3 (Factual correction response) is sent:**

- The template generates multiple outbound actions depending on outcome (applied / declined / additional docs requested).
- Each outcome is logged separately:
  - `[vendor-notification-sent-template-3]` for the response send itself
  - Include correction outcome in notes: applied / declined / additional-documentation-requested
  - For "applied" outcomes: include revision history anchor URL as additional artifact reference
  - For "declined" outcomes: include gap category applied (temporal mismatch / criterion mismatch / verifiability / scope mismatch)

**When Template 4 (Security team reply) is sent:**

- Log entry confirms length-symmetry principle compliance (the response was not longer than the incoming vendor message).
- Additional notes field: length ratio of response to incoming (roughly 1:1 or shorter is compliant).

**When vendor responses are received:**

- Per `#vendor-responses` inbound events matrix (Part 11), each response classification generates a specific event type.
- The log entry for the vendor response is separate from any Operations reply entry. Both are in `#vendor-responses`, same vendor identifier, sequential timestamps.

**When vendor correspondence escalates for internal review:**

- Log entry with `[vendor-response-legal-threat]` or `[vendor-proactive-contact]` (if substantive) event type, noting escalation timestamp and Strategy routing confirmation.
- Do not include any Operations response in the escalation log entry; there should not be an Operations response without Strategy approval (per vendor notification template Part 14 escalation paths).

- The QA REVIEW block output is saved to the Operations Project session log (not Discord).
- A Discord log entry is not required for routine QA CLEAR outcomes. The successful QA execution is embedded in the parent event's log entry (e.g., the vendor notification log entry that follows is evidence that QA CLEARED).
- For multi-template batch sends (e.g., Template 4 incident dispatch to 6 journalists), QA CLEAR is verified once and applies to all recipients; no per-recipient QA log is needed.

**When QA Protocol runs and fails (revision cycle triggered):**

- If QA identifies a CRITICAL issue requiring revision:
  - The revision cycle is internal (not logged to Discord) as long as resolution happens within the 2-cycle maximum.
  - Operations proceeds with revisions and re-runs QA until CLEAR.
- If QA CLEARS after revisions, proceed normally; no Discord entry for the revision process.

**When QA Protocol exhausts 2 revision cycles without resolving the CRITICAL issue:**

- The content is flagged `[UNRESOLVED — HUMAN REVIEW REQUIRED]` per QA Protocol Part 14 rules.
- Operations logs to `#daily-log` with `[qa-escalation-unresolved]` event type.
- Log entry fields: content type (vendor notification / media pitch / LinkedIn post / other), specific failure (Factual / Legal / Audience classification from Pass 1/2/3), Strategy routing timestamp, halted action.
- The unresolved content is not sent, not published, and not acted on until internal review reviews.

**When QA identifies a pattern across multiple content pieces:**

- If similar CRITICAL issues appear in 3+ different content pieces within a quarter (e.g., "Intent attribution to vendors keeps appearing in vendor notification drafts"), Operations logs a `#ideas` entry with `[pattern-observation]` event type.
- Include: pattern description, instances observed (with timestamps), Strategy review recommended.
- This is the pattern recognition function of Discord logs applied to QA itself, enabling meta-improvement of the template or training approach.

Across all five Operations documents, Discord logging integrates as the terminal step of each workflow:

**Workflow completion test:**

Operations can answer "is this task complete?" by checking two conditions:

1. The external action has executed successfully (email sent, post published, commit pushed).
2. The Discord log entry is posted in the correct channel with correct format.

Both conditions must be met for a task to be considered complete. Omitting either leaves the task in an incomplete state, regardless of external-action success.

**Single atomic unit:**

This positioning — logging as terminal workflow step, not as afterthought — means that Operations treats (external action + Discord log) as a single atomic unit. The cognitive frame is "I am executing Template X, which includes logging Y"; not "I executed Template X, and now I should remember to log Y."

The cognitive frame shift is what makes Discord logging reliable at scale. Without it, logging slips to "when I have time" and institutional memory degrades.

## Retention and privacy

Discord logs are long-term institutional records. Retention and privacy considerations determine how long logs remain accessible, who can see them, and what measures prevent unintended exposure. These considerations directly affect the three functions Discord logs serve (audit trail, institutional memory, pattern recognition) and VERDICT's legal posture.

The default posture is **indefinite retention with access controls**. Logs are not deleted as part of routine operations. When deletion is considered (legal requirement, privacy request, institutional decision), it follows the process defined in this section.

### Retention policy

**Default retention: indefinite**

VERDICT's Discord logs are retained indefinitely by default. Reasons:

- Audit trail function requires historical continuity; deleted logs create unretrievable gaps
- Institutional memory function accumulates value over time; early-phase logs become more valuable as context for later-phase patterns
- Pattern recognition function requires longitudinal data; short retention defeats pattern detection across years
- Legal discovery may require retrieval of historical logs; deletion can be challenged as spoliation

**Retention exceptions:**

The following circumstances may warrant deletion or redaction of specific logs:

- **Legal requirement:** A court order, regulatory notice, or data protection authority requirement explicitly mandating deletion. This is a internal review decision; Operations does not unilaterally delete logs in response to informal external requests.
- **Privacy request:** A journalist, vendor, or external party requests that specific log content be removed (typically for information they consider confidential that was inadvertently logged). Operations routes the request for internal review; only Strategy can approve redaction or deletion.
- **Accidental credential exposure:** Per Principle 5 and Exclusion rule 1, credentials must never be logged. If a credential is accidentally logged, the entry is redacted (credential removed, surrounding context preserved) immediately. This is the single exception where Operations acts without Strategy routing because security posture requires prompt response.

**Retention rationale in audit scenarios:**

If a future audit asks "why do you retain so much?" the institutional answer is:

- Logs are operational records, not personal data (credentials excluded, journalist / vendor professional correspondence properly categorized)
- Retention supports the three institutional functions of Discord logs
- No automated retention-limit expiration is applied; this is a deliberate institutional choice

This framing is consistent with rating agency practice: credit rating agencies retain historical rating actions and supporting analysis indefinitely because the historical record is the institutional product.

### Access control

**Default access scope:**

- Operations (operator) has full read and write access to all channels
- Co-founders and approved collaborators (when added) receive read access to all channels and write access to relevant channels (defined per role)
- internal review participants (currently operator alone; future: including co-founders) have read access to all channels

**Access additions:**

- New collaborator addition requires internal approval
- Access is granted via Discord server role assignments, not per-channel permissions (simpler management, consistent scope)
- Collaborator onboarding includes a review of this Discord logging rules document, particularly the content rules in Sections 4-5

**Access restrictions:**

- Discord server settings prevent messages from being forwarded outside the server
- Channel settings prevent non-member access
- Invitations to the server are closed; no public-link invites exist
- Screen sharing or screenshot of Discord logs requires operator approval for any external context

**Audit access:**

- External auditors (M&A, legal discovery) receive access through formal process: internal review reviews the request, determines scope, and grants time-limited read access to specific channels or all channels as appropriate
- Audit access logs the external party's presence for later review
- Auditor departure closes their access; no lingering access after audit completion

### Privacy considerations

**Personal data in logs:**

VERDICT's Discord logs contain professional correspondence (journalist publication bylines, vendor public contact aliases, platform names, etc.). These are not personal data in the sense protected by GDPR, CCPA, or similar regulations — they are professional identities.

However, certain content may cross into personal data territory:

- A journalist's personal email address (if accidentally discovered and logged)
- A vendor's personal mobile number (if unusually shared in correspondence)
- An individual's full name combined with role/location context that enables identification beyond their professional identity

**Privacy protective measures:**

- Per Exclusion rule 2 (Part 8), personal data beyond professional context is not logged
- If a personal email or phone is discovered in a correspondence, the professional alternative is logged instead (e.g., use published company email, not personal email)
- Sensitive context (vendor's claim about internal disagreement, journalist's tip about publication editorial decisions) is either summarized neutrally or pointed to the full text in a separate (restricted-access) location

**GDPR / CCPA compliance posture:**

- VERDICT operates as ZinovaCreation (Japan); GDPR applies to EU-facing data processing
- Discord logs contain minimal personal data by design; most content is corporate / platform / published-identity information
- For any data subject request (rights to access, erasure, etc.), internal review coordinates with legal counsel

**Privacy-by-design:**

- Default excludes personal data (Exclusion rule 2)
- Default uses professional identifiers (canonical forms from Part 6 Key identifier field)
- When personal data appears accidentally, logged entry is redacted and the error is itself logged (as `[governance-observation]` with `[credential-exposure-incident]` equivalent pattern)

### Screenshot and sharing discipline

**Internal sharing:**

- Discord logs may be shared via screenshot or text copy within VERDICT operational collaboration (e.g., between operator and co-founders when discussing a specific vendor situation)
- Shared screenshots should be cropped to the specific relevant entry; do not share full channel screenshots unnecessarily

**External sharing:**

- Discord log content is not shared externally (with journalists, vendors, investors, advisors) without internal approval
- This applies to both direct message sharing (sending a screenshot to an outside party) and indirect sharing (showing a screenshot in a meeting, conference, or presentation)
- Exception: Audit scenarios with Strategy-approved external auditor access per the access control rules above

### M&A and due diligence readiness

VERDICT's long-term trajectory may include M&A activity, investor review, or licensing transactions where Discord logs become part of institutional asset representation. Readiness for these scenarios is maintained structurally through the logging practices of this document.

**Why Discord logs are M&A-relevant:**

- Discord logs are operational evidence: they demonstrate that VERDICT operated with discipline over time
- Discord logs support valuation: consistent, structured logging indicates mature operations, which affects acquisition multiples
- Discord logs address acquirer questions: "how does VERDICT handle X situation?" can be answered with representative log entries rather than operational narrative alone
- Discord logs document independence: vendor correspondence logs, media pitch logs, and decision logs collectively demonstrate that VERDICT's independence claims are operationalized, not just declared

**Readiness signals maintained by logging discipline:**

- Channel discipline (events in correct channel) demonstrates operational rigor
- Format consistency (every entry with mandatory fields) indicates institutional sustainability
- Content quality (verbatim quotations, factual tone) shows legal-discovery readiness
- Retention without gaps demonstrates continuity
- Cross-reference traceability (logs ↔ documents) shows institutional coherence

**Preparing logs for due diligence:**

When M&A or investor review is anticipated:

- No special preparation of logs is needed; if logging discipline has been maintained, the logs are already due-diligence-ready
- If specific events need highlighting (e.g., a critical incident response), a summary memo can be prepared with log entry references; the logs themselves are not modified
- internal review coordinates with legal counsel on scope of log access provided to external parties

**What logs should NOT be used for in M&A contexts:**

- Logs are not marketing material; they are factual records
- Summaries of logs for investor presentations should be neutral ("Operations maintained consistent engagement with evaluated platforms") not promotional ("Operations executed flawlessly on every vendor notification")
- External sharing of raw log content requires Strategy approval per Part 16 access control

### Security incident response

If a security incident affects Discord logs — server compromise, account takeover, unauthorized access, credential leak within a log — response protocol follows:

**Incident detection:**

Incidents may be detected through:

- Discord security notifications (unusual login patterns, authorized IP changes)
- Operator noticing unfamiliar activity in channels
- External alert from Discord about server or account issues
- Internal discovery that sensitive content was inadvertently logged

**Immediate response:**

When incident is detected:

- **Rotate credentials immediately:** Discord account password, API tokens used by Operations Project session integrations, any other Discord-adjacent credentials
- **Preserve evidence:** Do NOT delete logs or channel history even if suspicious. Deletion creates additional legal exposure and destroys evidence needed for forensic review
- **Limit access:** Revoke access for any unfamiliar users or accounts; tighten server permissions
- **Document the incident:** Create a log entry in `#daily-log` with `[security-incident-detected]` event type (this is a novel event type that routes to `#daily-log` as default per safety valve; if this category becomes persistent, it should be added to the matrix formally)

**Incident documentation example:**

```
2026-04-23 14:30 JST | [security-incident-detected] | "Discord security incident" |
Type: Unusual login pattern detected by Discord security email |
Discovery time: 2026-04-23 14:15 JST (Discord email) |
Response actions: Password rotated immediately at 14:18 JST; 2FA re-verified at 14:20 JST; server member list reviewed at 14:25 JST — no unauthorized members |
Forensic preservation: All logs, channels, and message history preserved; no deletion performed |
Strategy routing: Incident reported for internal review at 14:28 JST |
→ internal review coordinates any external notification; Operations continues normal activity after credential rotation confirmed
```

**Escalation to Strategy:**

All security incidents handle per the internal escalation procedure immediately. Operations:

- Does not make external notifications unilaterally (e.g., does not notify journalists or vendors that a security incident occurred)
- Does not attempt forensic analysis beyond basic incident documentation
- Provides Strategy with complete incident context including log entries
- Awaits Strategy guidance on further steps (legal counsel consultation, external notification, Discord Trust & Safety engagement, etc.)

**Post-incident protocol:**

After incident resolution:

- Log entry with `[security-incident-resolved]` event type closes the incident
- Include: resolution summary, root cause if determined, preventive measures implemented
- If preventive measures suggest rule changes (e.g., "always use 2FA for all Discord accounts"), log to `#ideas` with `[operational-rule-created]` for document update consideration
- Maintain all incident logs indefinitely per retention policy; incidents are institutional learning evidence

**Credential leak within a log:**

If a credential appears in a log (per Exclusion rule 1 violation):

- **Redact immediately:** Edit the log entry to remove the credential, replacing with `[credential redacted 2026-MM-DD]` marker
- **Note the redaction:** Follow up with an additional log entry documenting the redaction (do not silently edit)
- **Rotate the credential:** The leaked credential is compromised by virtue of being logged in an accessible environment
- **Investigate the logging pattern:** Why was the credential logged? If it was a one-time error, note and proceed; if it was a recurring pattern, log to `#ideas` with `[governance-observation]` for Strategy review of training or template adjustment

### Anti-examples: common logging errors to avoid

The following examples demonstrate what NOT to do. Each anti-example violates specific rules from Parts 2 (Core principles), 6-7 (Format standards), or 8 (Content rules). Recognizing these patterns helps Operations avoid them in practice.

**Anti-example A: Missing mandatory fields**

```
Posted LinkedIn thing about aider. Got a few likes.
```

**Violations:**

- Missing timestamp (Part 6 Common structure)
- Missing event type (Part 6 Event type field)
- Missing key identifier (only casual reference "LinkedIn thing")
- Missing next action (Part 6 Next action field)
- Missing channel-specific additions (Part 7 `#daily-log` additions)
- Informal language ("thing," "a few likes") not institutional register

**Corrected version:**

```
2026-04-24 14:37 JST | [post-published-linkedin] | "LinkedIn Post #12 (aider evaluation)" |
Source: LinkedIn |
Artifact: linkedin.com/posts/zinova_aider-evaluation-activity-[post-id] |
First comment posted at 14:37:42 (42-second window, compliant) |
Outcome: success |
→ Monitor for 72h; no follow-up action required
```

**Anti-example B: Paraphrased vendor quote (Content rule violation)**

```
2026-04-26 16:42 JST | [vendor-response-substantive] | "Replit" |
Response classification: substantive-engagement |
Summary: Replit basically said our evaluation was unfair and their security posture is actually better than we reflected. |
→ Draft response
```

**Violations:**

- Paraphrased quote ("basically said," "our evaluation was unfair," "better than we reflected") rather than verbatim — violates Principle 4 and Exclusion rule 3
- Interpretive framing ("unfair") introduces bias — violates Exclusion rule 4
- Informal register ("basically said") not institutional — violates professional posture
- Vague next action ("Draft response") without time bounding — violates Part 6 Next action rules
- Missing full-text pointer (no vendor mailbox reference)

**Corrected version:**

```
2026-04-26 16:42 JST | [vendor-response-substantive] | "Replit" |
Response classification: substantive-engagement |
Received from: Juanma (Replit security team, security@replit.com) |
Key excerpt: "We appreciate the detailed evaluation and would like to note that our R-dimension score reflects our pre-2025 architecture. Our current architecture has addressed the specific concerns raised in Section 4.2 of the evaluation." |
Full text: vendor@getverdict.fyi mailbox thread #1287 |
→ Draft Template 4 response within 5 business days, apply length symmetry
```

**Anti-example C: Attack language (Content rule violation)**

```
2026-04-27 09:30 JST | [vendor-response-auto-reply] | "Manus AI" |
Manus AI predictably sent a generic auto-reply, which is exactly what you'd expect from a company that scored T:3/10 on transparency. Clearly they don't actually care about accountability. |
→ No response needed; their score speaks for itself
```

**Violations:**

- "Predictably" implies disdainful prediction — violates Exclusion rule 5 (unprofessional tone)
- "exactly what you'd expect" — interpretive framing
- "Clearly they don't actually care about accountability" — intent attribution (violates `qa_protocol_v1-1.md` Pass 2 Checklist applied through Exclusion rule 6)
- "their score speaks for itself" — dismissive language, attack-adjacent
- Overall tone: hostile characterization rather than factual record

**Corrected version:**

```
2026-04-27 09:30 JST | [vendor-response-auto-reply] | "Manus AI" |
Response classification: auto-reply |
Received from: support@manus.ai (automated acknowledgement) |
Content summary: Generic auto-reply acknowledging receipt, standard "we'll respond within X business days" template. No substantive engagement with evaluation content. |
Pattern note: Response pattern matches Manus AI precedent of T:3/10 transparency score; this is factual continuity, not pattern adjustment trigger. |
→ No response required (event concludes); continue monitoring for any substantive follow-up for 7 days
```

**Anti-example D: Credential exposure (Principle 5 violation — ABSOLUTE PROHIBITION)**

```
2026-04-28 11:00 JST | [operations-task] | "Site deployment" |
Deployed new evaluation page. Used SSH key with passphrase "verdict2026!" for the deployment script. Deployment successful at 11:02. |
→ Monitor for 24h
```

**Violations:**

- Passphrase logged in plaintext — violates Principle 5 and Exclusion rule 1, ABSOLUTE PROHIBITION
- Even the type of credential revealed (SSH key with passphrase) gives attacker context
- Event type `[operations-task]` is not a defined enum value; should be `[site-update-published]`
- Vague "Monitor for 24h" without specifying what to monitor

**Corrected version:**

```
2026-04-28 11:00 JST | [site-update-published] | "getverdict.fyi update (new aider evaluation page)" |
Source: Site repo (commit a7c3e91) |
Artifact: getverdict.fyi/aider/ |
Deployment: Automated via GitHub Actions, completed 11:02 JST |
Outcome: success (page live, index updated, no broken links) |
→ Monitor analytics for 7 days; no follow-up action required
```

**Credential note:** Authentication was via 1Password-managed SSH key; no credentials appear in log (per Principle 5).

**Anti-example E: Speculation as fact (Content rule violation)**

```
2026-04-29 14:00 JST | [pitch-sent-template-1] | "Thomas Claburn / The Register" |
Template used: 1 |
Subject: VERDICT evaluation: aider — independent AI agent trust data |
Thomas will probably write about this within the next week. He's known for quick coverage. |
→ Expect coverage by 2026-05-06
```

**Violations:**

- "Thomas will probably write" stated as prediction with unwarranted certainty — violates Exclusion rule 4
- "He's known for quick coverage" — speculation framed as known fact
- "Expect coverage by 2026-05-06" presents prediction as planned expectation
- Missing verification markers (when was recipient email verified? through what source?)

**Corrected version:**

```
2026-04-29 14:00 JST | [pitch-sent-template-1] | "Thomas Claburn / The Register" |
Template used: 1 (Cold pitch to journalist) |
Recipient: thomas.claburn@theregister.com (verified via The Register staff page, 2026-04-28) |
Subject: VERDICT evaluation: aider — independent AI agent trust data for security coverage |
Variant used: Standard opening |
Story hook: aider score 52/85 with zero CVEs in 12 months |
Expected response window: 14-21 days (per Template 1 timing rules) |
→ Monitor for response by 2026-05-20; if no response, consider Template 2 follow-up with new angle
```

### Pattern recognition through examples

The canonical examples (Parts 18-19) and anti-examples above illustrate the range of logging quality. Operations can internalize the structural patterns by reviewing examples periodically:

- **When drafting a new log entry type for the first time:** find the closest canonical example from Parts 18-19 and adapt structurally
- **When reviewing past logs during quality review (Part 9):** compare against canonical examples to identify drift
- **When onboarding a new collaborator:** the examples are the fastest way to convey institutional logging standards
- **When identifying a recurring error:** match the error against anti-examples to recognize the violation pattern

The examples are not exhaustive; they are representative. Novel event types or edge cases may not match any example exactly. In those cases, Operations extrapolates from the structural pattern closest to the situation, applies the Core principles and Content rules, and produces a compliant log entry.

### Compliance verification checklist

For periodic self-assessment of logging discipline (weekly quality review per Part 9):

- [ ] Every log entry has all four common mandatory fields (timestamp, event type, key identifier, next action)
- [ ] Channel-specific additions are present where applicable (Template used, Response classification, Citation flag, etc.)
- [ ] Timestamps are ISO 8601 with JST zone annotation
- [ ] Event types use defined enum values from the routing matrix (Parts 10-11) or `[other-flag-for-review]` if novel
- [ ] Key identifiers use canonical forms (platform name lowercase, journalist byline form, legal entity name for corporate events)
- [ ] External content is quoted verbatim (not paraphrased)
- [ ] No credentials appear in any log entry
- [ ] No attack language, intent attribution, or speculative predictions stated as facts
- [ ] Each entry is interpretable in isolation (sufficient context without requiring surrounding entries)
- [ ] Next action field explicitly states follow-up plan or `→ No action required (event concludes)`
- [ ] Cross-channel events generate separate entries in each affected channel (not single multi-channel entries)
- [ ] Escalations for internal review captured in original event's next-action field (no separate "routed" entry required)
